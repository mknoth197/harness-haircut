# HH-W012 — canonical source excluded by `.gitignore`

**Severity:** warn

## What fires it

The repo walker found that a canonical source — an `AGENTS.md` (at any
depth), or a directory on the `.agents/` path — is excluded by a rule in the
root `.gitignore`. Because git (and this tool) never collect an ignored path,
that content is **not** lifted into the IR.

Without this warning the exclusion would be silent: an over-broad rule such
as `*.md` or `.agents/` empties the canonical IR, and a subsequent `apply`
could then overwrite or delete provider files derived from *nothing*. HH-W012
exists so there is no silent loss of canonical sources the walk reaches: it
makes a reachable-but-excluded source visible so you decide deliberately.

Two shapes are reported:

- A single `AGENTS.md` path (e.g. `pkg/web/AGENTS.md`) — that file matched an
  ignore rule directly.
- A directory anchor ending in `/` (e.g. `.agents/` or `.agents/skills/`) —
  the directory was pruned, so every canonical file beneath it is lost. The
  walker reports the subtree rather than descending into an ignored tree
  (which would defeat the ignore rule).

## Scope / known limit

HH-W012 fires only for a canonical path the walk **reaches** and a **file-level**
rule excludes (a directly-ignored `AGENTS.md`, an ignored file under `.agents/`),
plus the pruned `.agents/` directory anchor itself.

A canonical file living **inside a directory you have fully gitignored** — for
example `vendor/AGENTS.md` or `pkg/AGENTS.md` when `.gitignore` contains
`vendor/` or `pkg/` — is **intentionally out of scope** and will **not** trigger
HH-W012. When you ignore a directory you have declared its entire contents
out of scope, so harness-haircut honors that directory ignore rather than
enumerating ignored trees to hunt for canonical files inside them — descending
into a user-ignored tree would defeat the very ignore you wrote. The walk
prunes the directory; nothing beneath it is collected and nothing is reported.

If you want such a file audited, un-ignore it (or the directory), or move it
out of the ignored tree.

## What to do

If the canonical source should be tracked, un-ignore it. Git's last-matching
pattern wins, so a negation line re-includes it:

```gitignore
*.md
!AGENTS.md
```

Note git's caveat: a negation **cannot** re-include a file whose parent
directory is excluded. If you ignored `.agents/`, a `!.agents/skills/foo.md`
line will not bring the file back — un-ignore the directory itself.

If the exclusion is intentional (the content is genuinely not canonical),
move it out of the canonical layout, or suppress the warning.

## Suppression

Add the code to `harness-haircut.config.json`:

```json
{ "suppressWarnings": ["HH-W012"] }
```
