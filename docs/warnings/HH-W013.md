# HH-W013 — provider path skipped: a symlink aliases it onto another repo path

**Severity:** warn

## What fires it

A provider projection target (for example `.claude/skills/<name>/SKILL.md`)
resolves through an **in-repo symlinked parent directory** onto a different
repo path. The typical shape is a hand-rolled consolidation that predates
harness-haircut:

```
.claude/skills/my-skill  ->  ../../.agents/skills/my-skill
```

Writing the projection "at" `.claude/skills/my-skill/SKILL.md` would follow
the symlink and land on `.agents/skills/my-skill/SKILL.md` — the canonical
source itself. Before this warning existed, `apply` did exactly that: the
canonical `SKILL.md` lost provider-stripped frontmatter keys
(`allowed-tools:`, `argument-hint:`), gained a `@generated` header, and the
very next `audit` reported the file as drifted — apply→audit broke its own
idempotency contract and destroyed canonical content (#35).

Both `audit` and `apply` now refuse to manage such a path: it is reported as
`aliased` (neither clean nor drift), never read for comparison, and never
written. The warning names the path and where it really resolves.

Only a symlinked **parent** triggers this. A symlinked *leaf* file at an
owned path is replaced in place with a real file by `apply` (the link, not
its target, is removed), which aliases nothing.

## What to do

Remove the symlink and let `apply` own the path as a real projection file:

```sh
rm .claude/skills/my-skill        # removes the LINK only; canonical content
harness-haircut apply             # writes a real .claude/skills/my-skill/
```

The canonical source under `.agents/` stays untouched throughout; the
projection is regenerated from it.

If the symlink is intentional and the provider should not be managed at all,
disable the provider in `harness-haircut.config.json` (e.g.
`{ "providers_disabled": ["claude"] }`).

## Suppression

Add the code to `harness-haircut.config.json`:

```json
{ "suppressWarnings": ["HH-W013"] }
```

Suppressing the warning does **not** make the path writable again — the skip
is a safety behavior, not a reporting choice.
